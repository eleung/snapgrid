# Core Concepts

A few ideas explain almost everything about how snapgrid behaves.

## The layout is controlled

snapgrid never holds your layout. You keep the array in state; snapgrid reads it, previews changes
while you drag, and hands you the next array through `onLayoutChange` when an interaction commits.

```tsx
const [layout, setLayout] = useState<Layout>(initial);
```

This means there's no hidden internal state to fall out of sync with, undo/redo is trivial (keep
your own history), and you can persist or transform the layout however you like. The trade-off is
that **you must apply the change**. If you don't call `setLayout`, nothing moves.

<Callout type="warning">
  Treat the layout as immutable. snapgrid clones internally and never mutates the array or items you
  pass in; you should do the same (always produce a new array in `onLayoutChange`).
</Callout>

## The layout model

A `Layout` is `LayoutItem[]`. Every item is positioned and sized in **grid units**, not pixels:

```ts
interface LayoutItem {
  i: string; // unique id â€” matches the child's React key
  x: number; // column (0-based)
  y: number; // row (0-based)
  w: number; // width in columns
  h: number; // height in rows
  minW?: number; maxW?: number; // resize limits
  minH?: number; maxH?: number;
  static?: boolean;     // never moves, never resizes; others flow around it
  isDraggable?: boolean; // per-item override of the grid default
  isResizable?: boolean;
  resizeHandles?: ResizeHandleAxis[]; // per-item handle set
}
```

Pixels come from the **`GridConfig`** (`cols`, `rowHeight`, `margin`, `containerPadding`,
`maxRows`) plus the measured container `width`. The geometry helpers turn `(x, y, w, h)` into
`left/top/width/height` and back.

## Built on dnd-kit

snapgrid is a grid layout built on **[dnd-kit](https://next.dndkit.com)** (the 0.4 rewrite). It does
not ship its own drag engine â€” it composes dnd-kit's. A grid lives inside a dnd-kit
`DragDropProvider`; tiles are dnd-kit draggables; the grid surface is a dnd-kit droppable; the drag
target comes from dnd-kit's collision system. snapgrid's job is the *grid* part on top: the
coordinate space, the packing, and the controlled layout.

That dependency is what makes a grid compose with the rest of the dnd-kit ecosystem â€” multiple grids,
sortable lists, and arbitrary draggables can share one provider and exchange items. The brains of
the layout (geometry, collision cascade, compaction) come from
[react-grid-layout](https://github.com/react-grid-layout/react-grid-layout)'s engine; the
*interaction* is dnd-kit.

## Headless first, with a turnkey layer

snapgrid is **headless-first**. The primary API is a set of hooks you wire to your own markup; the
`<GridLayout>` component is a thin, opinionated shell over those same hooks for the common case.

```tsx
// Headless â€” your markup, you own every element.
function Board({ layout, width, onLayoutChange }) {
  return (
    <DragDropProvider>            {/* dnd-kit's provider */}
      <Surface layout={layout} width={width} onLayoutChange={onLayoutChange} />
      <DragOverlay>{(source) => <Tile id={String(source.id)} />}</DragOverlay>
    </DragDropProvider>
  );
}

function Surface({ layout, width, onLayoutChange }) {
  // useGridContainer is the grid host: it owns the grid and returns its `group`.
  const { containerProps, group } = useGridContainer({ layout, width, onLayoutChange });
  return (
    <div {...containerProps}>
      {layout.map((it) => <Tile key={it.i} id={it.i} group={group} />)}
    </div>
  );
}

function Tile({ id, group }) {
  const { ref, style } = useGridItem(id, group); // resolves its grid by `group`
  return <div ref={ref} style={style}>{id}</div>;
}
```

```tsx
// Turnkey â€” the same thing, wired for you. <GridLayout> bundles the provider,
// the container, the items, the placeholder, and the overlay.
<GridLayout layout={layout} width={width} onLayoutChange={setLayout}>
  {layout.map((item) => <div key={item.i}>{item.i}</div>)}
</GridLayout>
```

<Callout type="info">
  **Two designs, on purpose.** The **headless** hooks are designed in dnd-kit's idiom â€” you bring a
  `DragDropProvider`, tiles declare a `group` the way [`useSortable`](https://next.dndkit.com)
  declares one, and grids interoperate with other dnd-kit draggables through `type`/`accept`. The
  **turnkey** `<GridLayout>` is designed in react-grid-layout's idiom â€” a controlled, drop-in
  component with `layout` / `onLayoutChange`. Reach for headless when you're composing with dnd-kit
  or need full control of the markup; reach for `<GridLayout>` when you want a grid that just works.
  They share one engine, so you can start turnkey and drop to hooks for any piece.
</Callout>

## How a tile finds its grid: `group`

In the headless API there is no per-grid React context â€” a tile resolves its grid by a **`group`**
key, exactly like a dnd-kit sortable item declares which list it belongs to. `useGridContainer`
returns the grid's `group` (its id); you pass it to each `useGridItem(id, group)`. This is what lets
several grids share one provider and still keep their items straight (so ids must be unique across
grids that share a provider).

## Compaction (packing)

After every move or resize, the layout is **re-packed** by a `Compactor`. The compactor decides
where items settle:

- `vertical` (default): items fall upward, like react-grid-layout.
- `horizontal`: items pack to the left.
- `none`: free positioning; items stay where you put them and may overlap.
- `masonry` / `gravity` / `shelf`: from `@snapgridjs/extras`, for variable-height packing.

You can swap the compactor at runtime (it's just an option) and even write your own. See
[Compaction & packing](/docs/guides/compaction).

## The interaction model

snapgrid leans on dnd-kit for the interaction and adds the grid math on top. While you drag:

- the dragged tile floats as a dnd-kit **`DragOverlay`** clone (so it can cross between grids
  unclipped), while the in-grid tile is hidden in place;
- dnd-kit's **collision** system resolves which grid the pointer is over (the drop target);
- snapgrid maps the pointer to a **cell** and re-packs, so a **placeholder** marks where the tile
  will land and other tiles animate to their reflowed positions in real time.

On drop, the previewed layout becomes the committed layout and flows back through `onLayoutChange`.
Lifecycle callbacks (`onDragStart`, `onDrag`, `onDragStop`, and the resize equivalents) fire
throughout. See [Events & lifecycle](/docs/guides/events).

<Callout type="info">
  Keyboard drags have no floating clone: the tile stays visible and steps cell-by-cell with the
  arrow keys (Enter/Space to pick up and drop, Escape to cancel).
</Callout>
